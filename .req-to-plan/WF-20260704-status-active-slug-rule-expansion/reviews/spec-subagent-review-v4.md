## Findings

### F1 [nit] case (d)'s "python 剔除" assertion is not observable through `detect()`
- Where / Issue: `SPEC-SYNCTEST-001` (d) asserts `ids.includes('fastapi') 且 !includes python 相关误取`. No stack keys on a `pythonDeps` value of `"python"`, and repo (d) has only `pyproject.toml` (no `.py`, so even the python language stack does not match), so a leaked `"python"` in the merged set would trigger no stack. The exclusion is correct by construction but has no concrete `detect()` observable — the second clause is effectively vacuous.
- Evidence: `parsePyproject` table branch `if (key && key != 'python' ...)`; no stacks.json entry has `pythonDeps:["python"]`.
- Suggested fix: Reduce (d)'s detect-level assertion to `ids.includes('fastapi')`, and if python-exclusion must be pinned, add a direct `mergePythonDeps` unit assertion that the returned set excludes `python` (or state it as covered-by-construction).

### F2 [nit] `parseRequirements` URL guard `tr.includes('://')` is over-broad
- Where / Issue: `parseRequirements` skips any line containing `://`. A legitimate dependency line carrying an inline URL comment (e.g. `requests==2.0  # see https://example.com`) would be dropped as a "URL line" (false negative). Low impact (best-effort, uncommon; no (a)–(g) fixture hits it).
- Evidence: `if (/^[a-z][a-z0-9+.-]*:\/\//i.test(tr) || tr.includes('://')) continue;`
- Suggested fix: Strip a trailing `#…` comment before the URL test, or gate the `://` check to the requirement token only, so an inline URL comment does not discard the dep.

### F3 [nit] malformed-pyproject fixture (f) doesn't exercise the `try/catch`
- Where / Issue: The (f) fixture `[[[bad` does not throw in `parsePyproject` — it is treated as section header `'[[[bad'` → `classify`='none' → inert. The asserted end-state (no crash, `vue` still detected) holds via robust parsing, not via the catch. Nothing in `parsePyproject` throws on string input, so the per-text `try/catch` is purely defensive (it mirrors the `package.json` `JSON.parse` precedent, which genuinely can throw).
- Evidence: trace of `parsePyproject('[[[bad', S)` — no throwing operation; `quotedStrings`/`stripInlineTables`/`split` cannot throw on a string.
- Suggested fix: Acceptable as-is (contract "malformed → no crash, others unaffected" is met). Optionally note in the SPEC that the pyproject catch is defensive-only, or keep (f) as a robustness (not exception-path) test.
