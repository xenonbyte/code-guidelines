## Findings

### F1 [severity: minor] DES-PIN-001 / SPEC-Handoff-#8 "exhaustive 48→57 list" omits three real `48` sites
- Where: DES-PIN-001, SPEC Handoff item 8; source `test/stacks.test.mjs:208` & `:213`, `test/precheck.test.mjs:68`, `AGENTS.md:12`.
- Issue: The design frames the propagation list as complete ("全部 48→57 联动位点清单"), but a repo-wide grep finds `48` references the list does not enumerate: (a) `test/stacks.test.mjs:208` test name `'every rules entry is one of the 48 stack ids'` and `:213` assertion string `'…is not one of the 48 stack ids'`; (b) `test/precheck.test.mjs:68` comment `// …without needing the 48-rule library`; (c) `AGENTS.md:12` documents `node --test --test-name-pattern="48"`, which is *coupled to the renamed count-test names* — if those test names go 48→57 but this pattern stays "48", the documented command silently matches zero tests (node --test exits 0 on no match). None of (a)–(c) turns a test red (they are strings/comments), so RISK-REGR-001's "漏改一处即测试红" mitigation still holds for the red-causing pins (both of which ARE covered). But they ship stale "48" into a 57-rule repo.
- Evidence: `grep -n 48` hits at the four locations above (verified); the design's DES-PIN-001 enumerates only library.test EXPECTED_FILE_COUNT+name, stacks.test length+name, README×2 (both langs), THIRD-PARTY/AGENTS/CLAUDE (generic), VERSION.
- Suggested fix: In DES-PIN-001 (and the SPEC checklist) explicitly add `test/stacks.test.mjs:208/213`, `test/precheck.test.mjs:68`, and call out `AGENTS.md:12`'s `--test-name-pattern` as coupled to the count-test renames (→ "57"), not merely a count digit. Note the generic "AGENTS.md 中 48→57" already covers AGENTS.md:12 *if* the implementer greps globally, but naming it removes the coupling trap.

### F3 [severity: minor] `mergePythonDeps` line-state-machine leaves two parsing cases undefined that can wrong-detect
- Where: DES-DETECT-002; SPEC Handoff item 1.
- Issue: The algorithm is sound in outline (array-mode for PEP621/PEP735 + optional-deps, table-key-mode for poetry excluding `python`, PEP508 head regex, PEP503 normalize) and correctly handles the cases the task flagged (poetry `python = "^3.11"` excluded; `foo = {version="1"}` table-key still yields `foo`; `uvicorn[standard]` → `uvicorn` via the `^[A-Za-z0-9][A-Za-z0-9._-]*` head-stop). But two cases are undefined and can misparse:
  1. **Single-line arrays.** The prose says "进入数组后逐行抽取引号内串，遇 `]` 结束" — silent on whether the *key line's own* post-`[` content is scanned. `dependencies = ["fastapi", "pydantic"]` (a very common inline form) would be *missed entirely* if extraction only starts on the next line — a false negative on the primary detection path (test (b)). SPEC Handoff #1 names "跨行数组终止" but not same-line entry extraction.
  2. **PEP 735 include-group inline tables.** `[dependency-groups]` arrays may contain `{include-group = "test"}` (an inline table, not a PEP508 string). "逐行抽取引号内串" would pull `test` and PEP508-parse it as a package name — a false dependency. Real-world collision risk is low (group names rarely equal a detected pythonDep) and it is best-effort, but it is genuinely undefined.
- Evidence: DES-DETECT-002 array-mode wording; PEP 735 include-group syntax; `[project].dependencies` is idiomatically written inline as often as multiline.
- Suggested fix: SPEC must state (i) array extraction scans the entry line's content after `[` (not only subsequent lines), covered by a single-line fixture in test (b) or a new case; (ii) inside dependency-groups arrays, skip tokens that are inline tables / the `include-group` key (or accept the bounded false-positive explicitly as a known limitation alongside SCOPE-OUT-002).
