- Suggested fix: Either drop it to "evidence" status and lean on the byte-for-byte SPEC compare (target 2), or use `node -e "require('...')"`-style parse only if a YAML parser is available; explicitly mark that authoritative validation is G1. (No functional impact — the file is deferred to CI anyway.)

### F2 [nit] TASK-003 verification hedges "readme.test.mjs（若涉及）"
- Where / Issue: T003 adds one CI sentence to `AGENTS.md`/`CLAUDE.md` and its verification says `node --test test/readme.test.mjs`（若涉及）. `readme.test.mjs` only guards `README.md`/`README.zh-CN.md` structure, not `AGENTS.md`/`CLAUDE.md`, so it is never relevant to T003 — the hedge is dead. Harmless but imprecise; the real gate is full `node --test` staying green (no test pins those two files' prose).
- Suggested fix: Replace with plain `node --test` (full suite green) + `git diff` evidence; drop the readme.test reference.

### F3 [nit] TASK-006 bundles 9 doc-heavy rule bodies with the count-pin mechanics — flag load, not a split
- Where / Issue: TASK-006 = 9 original rule files (each an official-doc + OWASP research-and-author unit) + 9 stacks entries + 16 pins + bilingual docs + VERSION, as one implementer/one reviewer unit. The count-pin/entry/file atomicity correctly forbids splitting the *entry+pin+count* mechanics. But the 9 rule *bodies* are mutually independent and independent of the pins; bundling ~9 doc-research sub-efforts under one task/one review risks shallow per-rule verification (RISK-SEC-002: wrong security guidance in `llm-app`/`electron`/`blazor`).
- Suggested fix: Keep it one commit (atomicity is real), but consider the rule bodies as a checklist of 9 independently-verified sub-items within the task (or a sibling authoring task feeding T006's single commit), so each rule's official-doc/OWASP verification is individually evidenced rather than rolled into one pass. Optional; does not change the commit boundary.
