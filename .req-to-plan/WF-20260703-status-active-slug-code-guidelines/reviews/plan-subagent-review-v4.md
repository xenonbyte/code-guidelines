## Findings (new or still-open)

### [MAJOR] TASK-009 stacks.test.mjs cross-file existence checks depend on files created only in TASK-013–018
- **File/where:** `07-plan.md` L212-213 (skeleton) and L217 (Step): the test asserts "rules 均指向存在的 assets/library 文件" and "lint 键(非 null 时)指向存在的 assets/lint/<lang> 目录" (Step: "rules→library、lint→baseline").
- **Failure scenario:** During the SDD loop, executing TASK-009 writes `assets/stacks.json` + `test/stacks.test.mjs`, then runs the gate `node --test test/stacks.test.mjs` (Verification L218). The assertions `existsSync('assets/library/<name>.md')` / `existsSync('assets/lint/<lang>')` are false because those files are created only in TASK-013–016 (library) and TASK-017–018 (lint), which run later. The gate fails in place → task cannot be verified at its position. This is the same forward-dependency class the v2 review flagged as MAJOR for old TASK-006/009; the sync.mjs-import fix resolved the original but the new "pure schema" wording overshot into on-disk resolution.
- **Fix:** Restrict TASK-009's stacks.test to genuine schema/shape validation (count=48, category enum, specificity int, four-predicate shape, rules/lint fields well-formed strings/arrays). Drop the rules→library and lint→baseline on-disk existence assertions from 009 — they already exist verbatim in TASK-021 (L480, L483), which runs after all 48 library files and 11 lint sets exist and is their correct home.
